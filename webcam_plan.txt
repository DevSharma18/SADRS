Simulation Overview
This plan describes how to build a fully functional local simulation of the SADRS CCTV Monitoring pipeline using your system's built-in webcam. The webcam replaces the IP camera that would be installed at an ATM in production. Everything else — ML behavioral detection, Claude Sonnet 4.6 AI analysis, simulated alerts, evidence clip saving, and a live dashboard — runs exactly as it would in the real deployment.

This simulation serves as Phase 2 of the SADRS implementation: proving the AI detection pipeline works before any hardware procurement or ATM deployment begins.

Simulation System Architecture
The simulation runs as a multi-threaded Python application. Four parallel threads handle different concerns simultaneously, just as they would on a Raspberry Pi in production. All threads share a central ThreatState object that acts as the source of truth for the current alert status.

Data Flow — Webcam to Alert
Below is the exact data flow from webcam pixel to alert dispatch. This flow is identical to what runs in production on a Raspberry Pi with a real ATM camera.

Step
What Happens
1
Webcam opens at 640×480, 15–30 FPS via OpenCV VideoCapture(0). Each frame is read into a NumPy array and timestamped.
2
Frame is resized to 640×640 and normalized (0–1 float32). Written into the 30-second rolling frame buffer (replaces pre-alert CCTV clip).
3
Frame is placed in the ML inference queue. Thread 2 picks it up and runs YOLOv8-nano TFLite inference. Output: list of (class_id, confidence, bbox) per frame.
4
ThreatEngine receives detection result. If class is in threat list (loiter, weapon, crowd, mask, aggressive): increment consecutive-frame counter. If counter ≥ 3: raise internal ThreatFlag.
5
ThreatEngine checks corroboration: did 'T' hotkey fire (tamper sim) OR did PIR sim trigger within the last 10 seconds? If yes → CONFIRMED THREAT. If no → flag held for 10 seconds pending corroboration.
6
AlertManager wakes on CONFIRMED THREAT event. Simultaneously: (a) extracts 30s pre-alert clip from ring buffer + 10s post-alert, saves encrypted .mp4 to /evidence/ folder. (b) Writes incident JSON to local log. (c) Activates on-screen LOCKDOWN overlay on the webcam window.
7
AlertManager calls Claude Sonnet 4.6 API with incident payload (threat class, confidence, timestamp, simulated sensor snapshot). Claude returns a 3-sentence incident summary.
8
Dashboard (Flask/SocketIO) receives incident push via WebSocket. Browser at localhost:5000 shows new incident card with Claude summary, clip link, threat class, confidence, and simulated GPS coordinates.
9
Console prints simulated GSM SMS text: 'ALERT: ATM-SIM-001 | Threat: {class} | Confidence: {score}% | GPS: {coords} | Time: {ts}'. This is exactly what the SIM800C would send in production.
10
Operator presses 'R' to reset alert state. ThreatEngine clears all flags. Incident marked as 'Awaiting Review' in dashboard. System resumes normal monitoring.

Detection Decision Logic
The threat engine applies the same multi-layer confirmation logic whether the source is a webcam or a real ATM camera. No shortcuts are taken in the simulation — this is intentional so that the same code can be deployed to production unchanged.

Logic Layer
Rule
Layer 1
Frame Confirm
Threat class must appear in ≥ 3 consecutive frames at ≥ 0.55 confidence. Single-frame detections are ignored entirely. This eliminates motion-blur false triggers.
Layer 2
Corroboration
CCTV threat flag must be accompanied by at least one corroborating signal within 10 seconds: 'T' hotkey (tamper sim), OR PIR simulation trigger, OR a second distinct threat class flagged simultaneously.
Layer 3
Time Weighting
Between 11 PM and 5 AM, the confidence threshold for Layer 1 drops from 0.55 to 0.45, and the corroboration window extends from 10 to 20 seconds. This makes the system more sensitive during high-risk hours.
Layer 4
Human Override
Bank operator can press 'D' to dismiss a confirmed threat (marks incident as false positive in log). Dismissed incidents feed the model retraining pipeline. Operator can also press 'E' to escalate, which upgrades alert to Priority 1.
Layer 5
Cooldown
After an alert fires, a 120-second cooldown prevents repeated alerts for the same ongoing event. Cooldown can be manually reset with 'R'. New distinct threat classes bypass the cooldown.
